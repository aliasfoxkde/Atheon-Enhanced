What it is

Atheon is a pattern matching engine. You define what you're looking for. You point it at a file, a directory, your environment variables, or a stream of text. It finds every match and tells you exactly where.

The engine doesn't have opinions about what matters. You do — so you write the rules, and Atheon enforces them. Over anything.

Why it matters

Sensitive data ends up where it shouldn't — constantly. A hardcoded credential in a config file. A token accidentally printed to a log. A secret committed to a repository and now permanently in history.

These aren't rare edge cases. They happen to careful teams running good processes. The difference between a near-miss and a breach is often just whether anyone was looking.

Atheon is the thing that's always looking.

The scenario

A developer pushes a configuration file. Buried in a comment from a debugging session weeks ago is a production API key. The commit goes through. The pipeline passes. Two months later, someone notices unusual billing activity.

Atheon, wired into a pre-push hook:

$ atheon ./

[api-key] config/app.yaml:47  →  # debug key: sk-prod-a8f3c...

Exit code 1. The push never happens. The key never leaves the machine. The incident never occurs.

Infinitely extensible

A pattern is just a name and a match function. That's the entire contract. It can be a regex, a keyword, a structural check — anything that returns true or false on a line of text.

That simplicity is intentional. There's no config format to learn, no DSL, no plugin manager. You write Go, you drop the file in, you rebuild. The pattern appears immediately in atheon list. The engine doesn't change. Only your rules grow.

Security teams use it to catch leaked credentials. Compliance teams use it to scan for PII. Legal teams search contracts for specific clauses. Operations teams monitor pipeline logs. Healthcare, finance, internal tooling — if the domain has text and rules, Atheon runs those rules.

This is the beginning of what it can become.
